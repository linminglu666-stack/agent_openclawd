## 计划原文摘录
1) 机制目标与适用范围
目标

可控性：你随时知道“做到哪、产出了什么、还剩什么”。

可验收性：每次汇报都带可核验产物（结论段落、结构草案、表格片段、清单等）。

低噪音：只在模块完成时汇报，避免无效刷屏。

适用范围

分析/方案/改稿/文档整理/复杂推理/多文件处理/批量任务/多版本输出等。

单步回答（不需要拆分）可不启用，但你要求“输出完整方案”这种就默认启用。

2) 核心规则（硬约束）
R1：模块化先行

在执行前，必须先给出模块划分（或在心里完成划分后直接按模块推进），每个模块要具备：

模块名

产出物

验收标准（你一眼能判断“这模块算完成了吗？”）

模块是“可独立验收的一段工作”，不是过程动作（例如“思考一下”不算模块）。

R2：每完成一个模块，立即汇报一次

汇报内容必须包含：本模块产出（可直接看/用）+ 下一模块计划 + 风险/依赖（如有）。

不做“我正在努力”的叙述型汇报，只做结果型汇报。

R3：任务完成后，立即给完成汇报

完成汇报必须包含：

交付物清单

结果摘要（要点）

关键假设/边界

下一步建议（最多3条，按性价比排序）

3) 模块定义标准（如何拆才“对”）
3.1 模块粒度准则

优先选择“每个模块产出一个可用中间件”的粒度：

方案类：结构大纲 → 核心方案 → 风险校验 → 定稿与可执行清单

分析类：输入抽取 → 框架/指标 → 推理与结论 → 验证与反例 → 总结

多文件类：文件分组与索引 → 逐组结论 → 跨文件一致性 → 汇总输出

避免过细：比如把“写一段话”拆成3个模块会造成节奏碎片化。
避免过粗：把“完整方案”当成一个模块会失去进度可视性。

3.2 模块验收标准（建议固定三类）

每个模块至少满足其中一种：

可读：一段可直接阅读的结论/草案/摘要

可操作：一份可执行清单/规则/参数表

可复核：引用了输入依据或明确说明“依据来自哪里/假设是什么”

4) 标准工作流（从接单到交付）
阶段 A：任务建模（模块计划）

输出（或内部确定）模块列表（N个）

明确“完成态长什么样”（最终交付物形态：文档/清单/表格/对比方案等）

如果你未提供足够输入：仍然先推进可确定模块（比如“约束抽取、风险列举、默认假设”），在汇报里明确缺口与影响。

阶段 B：按模块执行

逐模块产出

每完成一个模块立即汇报（见第5节模板）

阶段 C：收束与定稿

合并所有模块产出为最终交付

立刻输出完成汇报（见第5节模板）

5) 汇报模板（固定格式，便于你扫读）
5.1 模块完成·进度汇报模板（必用）

【进度｜模块完成】M<i>/<N>：<模块名>

本模块产出：<直接可用的内容/结论/清单（核心放这里）>

验收点：<一句话说明为什么算完成>

下一模块：<模块名 + 准备做什么>

风险/依赖：<无 / 具体风险 + 影响 + 缓解策略>

规则：产出必须可见（至少给关键段落/清单，不只说“已完成”）。

5.2 完成·结果汇报模板（必用）

【完成｜结果总结】

交付物清单：

…（用途/怎么用）

…

结果摘要（要点 5–10 条）：

…

关键假设/边界：

…

下一步建议（≤3条）：

…

…

6) 异常与阻塞处理（必须“即时汇报”）

出现以下情况，不等模块完成，立即插入一次 【受阻｜快照】 汇报：

输入缺失/歧义导致无法继续（例如关键参数未给）

文件无法读取/过大/格式损坏

发现前提冲突（A与B同时为真会导致方案失效）

【受阻｜快照】模板

受阻点：是什么卡住了

已确认事实：当前能确定的部分（避免“完全停摆”）

影响范围：会影响哪些模块/交付物

解决路径：给出 1–3 个选项（含默认推荐）

7) 大文件/长内容的拆分规则（你之前的要求纳入机制）

当输入文件过大或内容过长时：

先给拆分方案（按章节/主题/时间线/模块）

边拆边产出：每完成一个拆分单元视作一个模块并汇报

最终再做跨单元汇总与一致性校验

拆分建议最小信息集：

拆分维度（章节/主题/模块）

每块预期产出（提要/结论/问题清单）

汇总方式（统一结论表/冲突对照表/优先级清单）

8) 质量门槛（每个模块都要过的“自检”）

每次模块完成汇报前，至少自检 4 项：

范围：有没有跑题/越界输出你不需要的内容

依据：结论是否有输入依据或明确假设

可用：你能否直接复制执行/直接评审

一致性：与前一模块产出是否冲突（如冲突必须标注）
